# 飞书数据源联动版设计

## 目标

把当前临时 SQLite 内测版升级为飞书联动版。网站继续负责低门槛使用和后台审核，飞书负责团队协作、长期沉淀和结构化数据保存。

这版要让团队成员不需要理解数据库、GitHub 或部署细节，只要打开网站就能查资产、投稿、复用；管理员在后台审核、去重、更新和发布；飞书知识库保留每条资产的详情页、案例、附件和多人协作记录。

## 推荐架构

使用“飞书多维表格 + 飞书知识库详情页”的组合。

- 飞书多维表格做结构化数据库，保存列表、状态、审核、去重、更新时间等字段。
- 飞书知识库做详情页，承载长文 SOP、使用案例、截图、附件和讨论。
- 现有网站前台读取正式资产和投稿状态。
- 现有网站后台处理审核、重复合并、发布、退回和备份。
- 现有 SQLite 仅保留为本地开发和飞书不可达时的兜底，不再作为线上正式数据源。

## 飞书表结构

第一版需要 4 张表。

### 工作流资产表

正式资产库，只展示审核通过的资产。

核心字段：

- 资产ID
- 资产标题
- 主分类
- 适用岗位
- 解决痛点
- 工具或链接
- 知识库详情页
- 输入材料
- 使用步骤
- 输出结果
- 失败处理
- 人工审核边界
- 是否可生成 Skill
- 是否包含敏感信息
- 审核状态
- 贡献人
- 审核人
- 更新时间
- 版本号

### 投稿审核池

团队成员提交的新资产先进入这里。

核心字段：

- 投稿ID
- 资产标题
- 投稿人
- 适用岗位
- 主分类
- 一句话说明
- 工具或链接
- 原始投稿内容
- 敏感信息标记
- 重复候选
- 重复处理方式
- 目标资产ID
- 审核状态
- 创建时间
- 更新时间

### 重复与更新记录

记录投稿和已有资产之间的关系，防止越积越乱。

核心字段：

- 记录ID
- 投稿ID
- 目标资产ID
- 相似原因
- 处理方式
- 处理人
- 处理时间
- 备注

处理方式：

- 新资产
- 补充已有资产
- 更新已有资产
- 拒绝重复投稿

### 审核日志与管理员表

记录谁做了什么，保留团队交叉审核机制。

核心字段：

- 记录ID
- 操作类型
- 操作对象ID
- 操作人
- 操作结果
- 操作时间
- 备注

管理员字段：

- 姓名
- 角色
- 权限
- 状态

## 数据流

### 前台读取

前台启动时读取飞书工作流资产表，只展示审核通过、未隐藏的正式资产。

如果飞书 API 不可达，页面显示“飞书数据暂时不可用”，并保留本地种子资产作为演示兜底，但不能把兜底数据误报为最新数据。

### 投稿

团队成员在前台提交投稿后，后端先读取正式资产表和投稿审核池，做一次重复检查。

如果没有明显重复，写入投稿审核池，状态为待审核。

如果可能重复，写入投稿审核池时附带重复候选，并提示这次投稿更适合走“补充已有资产 / 更新已有资产”。

### 后台审核

管理员后台读取投稿审核池、正式资产表、重复记录和审核日志。

管理员可以执行：

- 发布为新资产
- 补充已有资产
- 更新已有资产
- 标记需补充
- 退回

发布或更新时同步写入工作流资产表，并写入审核日志。

### 知识库详情页

每条正式资产可以关联一个飞书知识库详情页。详情页承载长文，不强行塞进网站表格字段。

短期可以人工创建详情页并把链接填入表格。后续再考虑自动创建详情页。

## 权限边界

不要让所有人直接编辑正式资产表。

建议权限：

- 普通成员：可看正式资产，可提交投稿，可编辑自己的草稿或知识库补充。
- 审核员：可审核、退回、标记需补充、维护重复关系。
- 超级管理员：可发布、合并、下架、导出和修改管理员配置。
- 老板/团队老大：可查看全局、看审核状态、做最终判断。

如果飞书空间必须设置成大家可编辑，则网站后台仍应把正式发布动作收口到管理员，避免绕过审核。

## 环境变量

新增飞书数据源配置：

- `DATA_SOURCE=feishu`
- `FEISHU_APP_ID`
- `FEISHU_APP_SECRET`
- `FEISHU_BASE_APP_TOKEN`
- `FEISHU_ASSETS_TABLE_ID`
- `FEISHU_SUBMISSIONS_TABLE_ID`
- `FEISHU_DUPLICATES_TABLE_ID`
- `FEISHU_AUDIT_TABLE_ID`
- `FEISHU_WIKI_HOME_URL`

保留：

- `ADMIN_TOKEN`
- `OPS_DB_PATH`

当 `DATA_SOURCE=sqlite` 或飞书配置缺失时，系统继续使用 SQLite，方便本地开发和回滚。

## 错误处理

- 飞书网络不可达：明确提示网络或飞书 API 暂不可用，不调试字段。
- 飞书凭证错误：提示需要检查 `FEISHU_APP_ID` / `FEISHU_APP_SECRET`。
- 表格字段缺失：启动时返回字段检查结果，列出缺失字段。
- 投稿写入失败：不假装成功，前台显示需要重试。
- 审核写入失败：后台保留当前页面状态，并提示没有发布成功。

## 验收

- 前台能从飞书读取正式资产。
- 前台投稿能写入飞书投稿审核池。
- 重复检查同时检查正式资产和待审核池。
- 后台能读取飞书投稿池并执行发布、退回、需补充。
- 发布后前台刷新能看到新资产。
- 每次发布、退回、更新都会写入审核日志。
- 飞书不可达时，页面有清楚提示，不把旧数据当成最新数据。
- 原有 SQLite 测试继续保留，新增飞书数据源测试使用 mock，不依赖真实飞书网络。

## 不做范围

第一版不做飞书账号单点登录，不自动创建知识库页面，不做复杂版本 diff，不做企业级权限系统。

第一版重点是把“前台投稿 -> 后台审核去重 -> 发布正式资产 -> 飞书长期保存”的闭环跑通。
