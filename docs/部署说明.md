# 部署说明

## 目标

第一阶段部署成内部内测版：同事打开一个 URL 可以提交投稿，后端保存到 SQLite，管理员审核通过后发布为正式资产。

## M1 推荐架构

```text
最快内测：一个 Render Web Service
  同一个 URL 同时提供前端页面和后端 API

后续扩展：GitHub Pages / Vercel 静态前端 + Render / Railway / Fly.io 后端
  适合内测稳定后再拆分
```

M1 先推荐单服务，是因为同事只需要打开一个 URL，不需要理解“前端地址”和“后端地址”的区别。

## Render 一键部署准备

项目根目录已经提供：

```text
render.yaml
```

它会创建一个 Node Web Service，并把 SQLite 写到持久盘：

```text
OPS_DB_PATH=/opt/render/project/src/storage/ops-assets.sqlite
```

Render 官方文档说明：Blueprint 默认使用仓库根目录的 `render.yaml`；持久盘需要挂到绝对路径，只有挂载路径下的数据会在部署和重启后保留。

## Render 手动步骤

1. 在 GitHub 新建并推送这个仓库。
2. 打开 Render，选择 New Blueprint。
3. 选择这个 GitHub 仓库。
4. 确认 `render.yaml` 被识别。
5. 设置 `ADMIN_TOKEN`，这是管理员发布资产时用的审核密码。
6. 部署完成后打开：

```text
https://ops-codex-workflow-library.onrender.com/
```

7. 检查健康接口：

```text
https://ops-codex-workflow-library.onrender.com/api/health
```

如果 Render 分配的域名不是上面这个，需要同步修改：

```text
PUBLIC_ORIGINS=你的实际 Render 域名
```

## 后端环境变量

```text
PORT=8787
ADMIN_TOKEN=只给管理员使用的审核密码
PUBLIC_ORIGINS=https://你的前端域名
OPS_DB_PATH=/data/ops-assets.sqlite
```

本项目 Render 单服务默认使用：

```text
OPS_DB_PATH=/opt/render/project/src/storage/ops-assets.sqlite
PUBLIC_API_BASE=same-origin
```

## 后端健康检查

```text
GET /api/health
```

成功时返回：

```json
{
  "ok": true,
  "mode": "realtime-backend"
}
```

## 前端接入云后端

如果是单服务部署，不需要改页面，`apps/web/config.js` 默认使用：

```text
same-origin
```

如果是 GitHub Pages / Vercel 静态前端，修改或生成 `apps/web/config.js`：

```js
window.OPS_CONFIG = {
  "apiBase": "https://你的后端服务地址"
}
```

也可以用命令生成：

```bash
PUBLIC_API_BASE=https://你的后端服务地址 node scripts/write-web-config.mjs
```

本地访问 `127.0.0.1` 或 `localhost` 时，前端仍默认连接 `http://127.0.0.1:8787`。

## 数据持久化

云平台必须配置持久磁盘，并把 `OPS_DB_PATH` 指向持久磁盘路径。不要只使用临时容器目录。

Render 持久盘建议：

```text
mountPath=/opt/render/project/src/storage
sizeGB=1
```

## 备份

最低要求：

- 发布内测前手动备份一次。
- 每天自动备份一次 SQLite 文件或确认云平台持久盘快照策略。
- 每次管理员批量发布前导出正式资产。

本地备份命令：

```bash
node scripts/backup-sqlite.mjs
```

本地导出命令：

```bash
node scripts/export-assets.mjs
```

Render 的持久盘会有平台快照，但正式资产仍建议定期导出，避免只依赖整盘恢复。
